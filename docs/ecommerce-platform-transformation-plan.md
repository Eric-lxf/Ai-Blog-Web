# 电商平台化改造计划：从 Ai-Blog-Web 到完整 C 端购物平台

**状态**：Draft
**作者**：Cloud Agent（架构 & 产品视角）
**最后更新**：2026-07-17
**范围**：后台系统全景梳理 + 现有代码库改造映射 + 分期路线图 + 各期规模分级与验收标准

## 1. 背景与现状

### 1.1 当前项目是什么

`Ai-Blog-Web` 本质是 **RuoYi-Vue 3.9.2 管理后台骨架 + 一个业务模块（博客）**，不是电商系统。现状盘点：

| 层 | 现状 | 证据 |
|---|---|---|
| 后台基座 | RBAC、菜单、字典、日志、定时任务、代码生成器 | `backend/ruoyi-system`、`ruoyi-framework`、`ruoyi-quartz`、`ruoyi-generator` |
| 业务模块 | 博客（文章/分类/标签/评论/审核/通知/账单识别/AI 写作） | `backend/ruoyi-blog/.../controller`：`BlogArticleController`、`BlogCommentController`、`BlogBillController` 等 |
| 渠道模块 | 微信公众号/小程序对接 | `backend/ruoyi-wechat` |
| AI 能力 | 多 Provider 配置（OpenAI/Claude/DeepSeek）、AI 写作/优化 | `AiProviderController`、`AiWriteController`、`AiOptimizeController` |
| 前端 | 管理后台（Element Plus）+ 博客 C 端页面 `/blog` | `frontend/src/views/system`、`frontend/src/views/public/blog` |
| 数据 | MySQL `ai_blog` + Redis（Token/验证码） | `sql/*.sql` |

**关键结论**：当前代码库里**完全没有**商品、库存、订单、购物车、支付、物流、营销这些电商核心域。改造不是"加功能"，而是在现有后台基座（RBAC/日志/定时任务/文件上传/AI能力/微信渠道）之上，**新增一整套交易域业务模块**，并新建一个独立的 C 端商城前台。

### 1.2 目标

把项目改造成一个「淘宝/京东/Shopify/SHEIN/拼多多」量级参照下的**自营为主**电商平台（多商家入驻能力作为可选后续阶段），包含：
- C 端购物网站（用户看到的部分，工作量占比小）
- 完整后台系统群（商品、交易、库存、支付、营销、物流、客服、风控、数据……工作量占比大，本文重点）

## 2. 成熟电商平台的后台系统全景

参考淘宝/京东/Shopify/SHEIN/拼多多的公开架构资料与行业通识，一个"完整"的电商后台由以下 **4 大域、约 18 个子系统**构成。C 端网站只是这些系统能力的一个"出口"。

### 2.1 交易核心域（决定"能不能把货卖出去"）

| 系统 | 核心职责 | 典型实体/能力 | 对标 |
|---|---|---|---|
| 用户/会员中心 | 注册登录、账号安全、收货地址、会员等级、积分、第三方登录 | User、Address、MemberLevel、Points | 淘宝会员体系、京东PLUS |
| 商品中心（PIM） | 类目、品牌、SPU/SKU、属性、图片视频、多规格、商品审核 | Category、Brand、SPU、SKU、Attribute | 京东商品中心、SHEIN 多规格 SKU |
| 购物车/订单中心（OMS） | 加购、下单、订单状态机、拆单合单、改价、取消 | Cart、Order、OrderItem、OrderStateMachine | 淘宝交易系统 |
| 库存中心（WMS 前置） | 多仓库存、预占/锁定/扣减、安全库存、超卖防护 | Inventory、StockLock、Warehouse | 京东自建仓库存 |
| 支付中心 | 收银台、支付渠道网关、异步回调、对账、退款、分账 | PaymentOrder、Channel（微信支付/支付宝/Stripe） | 支付宝/微信支付网关模式 |
| 物流中心（TMS） | 运费模板、承运商对接、发货、物流轨迹、逆向物流 | Shipment、Carrier、Tracking | 菜鸟/京东物流 |
| 售后/逆向中心 | 退货/退款/换货审核、逆向物流、纠纷仲裁 | AfterSaleOrder、RefundOrder | 淘宝售后中心 |

### 2.2 商品与供应链域（决定"货从哪来"）

| 系统 | 核心职责 | 对标 |
|---|---|---|
| 供应链/采购中心 | 供应商管理、采购单、入库、成本核算 | 京东自营供应链 |
| 商家/店铺中心（如做平台型） | 商家入驻审核、店铺装修、多商家商品发布、结算分账 | 淘宝商家后台、Shopify Admin |
| 仓储管理（WMS） | 分仓策略、拣货/打包/出库、库内作业 | 菜鸟仓配 |

> 若定位为**自营电商**（类似京东自营/SHEIN 一开始），供应链/商家中心可大幅简化甚至不做；若要做**平台型**（类似淘宝/拼多多），商家中心是必选项。这是需要业务方明确的关键决策，见第 9 节。

### 2.3 营销与内容域（决定"用户为什么买、买得多"）

| 系统 | 核心职责 | 对标 |
|---|---|---|
| 营销中心 | 优惠券、满减满赠、限时折扣、秒杀、拼团、会员日 | 拼多多拼团、淘宝双11玩法 |
| 搜索与推荐 | 商品搜索引擎、个性化推荐、类目导航、搜索排序 | 淘宝搜索、SHEIN 个性化 |
| 内容运营（CMS） | 首页楼层、专题页、Banner、图文详情、直播/短视频入口 | 京东首页运营位 |
| 评价/UGC | 商品评价、晒图、问大家、评价审核（可复用现有评论/敏感词能力） | 淘宝评价体系 |
| 消息与通知中心 | 站内信、短信、邮件、Push、微信模板消息（统一出口） | 各平台消息中台 |

### 2.4 支撑与治理域（决定"能不能稳定、安全地长期运营"）

| 系统 | 核心职责 | 对标 |
|---|---|---|
| 风控与安全中心 | 反刷单、反欺诈、设备指纹、支付风控、黑名单 | 支付宝风控引擎 |
| 客服中心 | 工单系统、在线 IM 客服、售后接入 | 淘宝千牛客服 |
| 财务结算中心 | 商家结算、发票、退款流水、对账报表 | 京东财务中台 |
| 数据中台/BI | 用户行为埋点、经营看板、A/B 测试、实时大屏 | 淘宝生意参谋 |
| 权限与运营后台基础设施 | RBAC、菜单、日志、字典、定时任务、代码生成 | **当前项目已具备（RuoYi）** |
| 开放平台/API 网关 | 多端接入（App/小程序/开放API）、限流鉴权 | 淘宝开放平台 |

**小结**：C 端购物网站在这 18 个系统里只对应"商品展示、购物车、下单、支付跳转、订单查询、评价查看"这几个**读多写少的前台交互**；剩下绝大部分系统（库存、履约、营销引擎、风控、结算、数据）全部在后台，这也是用户判断"工作量集中在后台"的依据。

## 3. 现状 → 目标：系统映射与改造方式

| 目标系统 | 处理方式 | 说明 |
|---|---|---|
| RBAC/日志/字典/定时任务/代码生成 | **直接复用** | `ruoyi-system`/`ruoyi-framework`/`ruoyi-quartz`/`ruoyi-generator` 原样保留，作为所有新模块的基座 |
| 文件上传（图片/视频） | **直接复用** | `BlogFileController`/`FileUploadController` 改造为商品图片/视频上传的通用能力 |
| AI 能力（多 Provider） | **复用并延伸** | 用于：商品文案/详情页生成、智能客服问答、评价情感分析、选品建议 |
| 微信模块 `ruoyi-wechat` | **复用并延伸** | 用于：微信支付基础对接、小程序商城渠道、模板消息（发货/退款通知） |
| 博客模块 `ruoyi-blog` | **保留不动或降级为内容子系统** | 博客与电商是两个业务域，不建议强行合并进商品模块；可作为"内容营销/SEO引流"子系统保留，或后续独立拆分 |
| 评论/敏感词/通知能力 | **抽取为公共能力** | `BlogComment*`、`BlogSensitiveWord*`、`BlogNotification*` 的审核/敏感词/通知机制可下沉为公共组件，供"商品评价"复用，避免重复造轮子 |
| 商品/库存/订单/支付/物流/营销/售后/风控/结算/搜索/客服 | **全部新建** | 当前代码库中不存在，是本次改造的主体工作量 |
| C 端商城前台 | **新建**，与现有 `/blog` C 端页面分离 | 见第 6 节 |

## 4. 分期路线图（按技术范围分期，不按日历时间）

> 遵循"先打通最小闭环，再补齐履约与运营能力，最后平台化"的演进式架构原则，每期都是可独立上线验证的里程碑。
>
> **关于"预计周期"的说明**：本文不给出具体天数/周数的日历估算——团队人力、并行度、AI 辅助编码效率差异很大，日历估算在这个阶段大概率是"看起来精确但实际不准"的假数字。取而代之，用**规模分级（S/M/L/XL）**刻画每期的真实复杂度，判据是：①新增模块/表数量 ②是否引入新中间件（ES/MQ/分布式锁）③涉及的外部系统对接数（支付渠道、物流承运商）④一致性/并发风险等级。规模分级可直接换算成排期输入：给定团队人力后，由研发负责人按 S/M/L/XL 结合自身速率二次估算天数，而不是我在这里假设一个团队规模。

### 🟢 Phase 1 — 交易最小闭环（MVP）

**目标**：用户能在 C 端浏览商品、加购、下单、完成支付，运营能在后台管理商品和订单。

**规模：L**（4 个新模块、约 10+ 张新表、1 类新外部依赖——支付渠道对接；无新中间件，风险等级中——涉及资金流但不涉及高并发超卖场景）

| 新增模块 | 核心内容 |
|---|---|
| `ruoyi-mall-product`（商品中心） | 类目树、品牌、SPU/SKU、多规格、商品图片、上下架审核 |
| `ruoyi-mall-order`（交易中心） | 购物车、下单、订单状态机（待支付→待发货→待收货→完成/取消）、订单详情 |
| `ruoyi-mall-payment`（支付中心） | 微信支付/支付宝下单接口、异步回调、支付状态同步、简单对账 |
| 会员体系扩展 | 现有用户体系 + 收货地址簿 |
| C 端商城前台 v1 | 首页、类目/列表页、商品详情、购物车、结算页、订单查询 |

**验收标准**：
- [ ] Given 后台已创建商品（含至少一个已上架 SKU），When 运营在商品中心查看列表，Then 能看到该 SKU 的价格/库存/上下架状态
- [ ] Given C 端用户已登录，When 在商品详情页点击加入购物车并结算下单，Then 生成一条状态为「待支付」的订单，且金额与商品价格×数量一致
- [ ] Given 用户在结算页选择微信支付/支付宝并完成支付，When 支付渠道异步回调到达，Then 订单状态自动变为「待发货」，且重复回调不会导致重复扣款或状态错乱（幂等性）
- [ ] Given 订单已支付，When 用户在"订单中心"查看，Then 能看到订单详情、金额、状态与商品快照（即使后续商品信息被修改，订单里的商品信息不变）
- [ ] Given 用户下单后 30 分钟未支付，When 定时任务扫描超时订单，Then 订单自动关闭为「已取消」
- [ ] 后台运营可按状态/时间筛选订单列表，并可查看单条订单的完整状态流转日志

### 🟡 Phase 2 — 完整履约闭环

**规模：XL**（4 个新模块、库存中心涉及并发安全设计——最高风险项、物流需对接至少 1 家外部快递查询服务、售后涉及资金退款与逆向流程）

| 新增模块 | 核心内容 |
|---|---|
| `ruoyi-mall-inventory`（库存中心） | 多仓库存、下单预占/支付扣减、Redis+Lua 防超卖、安全库存告警 |
| `ruoyi-mall-logistics`（物流中心） | 运费模板、发货单、对接快递查询（快递100/菜鸟）、物流轨迹展示 |
| `ruoyi-mall-aftersale`（售后中心） | 退货/退款/换货申请与审核、逆向物流单 |
| `ruoyi-mall-review`（评价系统） | 商品评价、晒图、复用博客模块的敏感词/审核能力 |

**验收标准**：
- [ ] Given 某 SKU 库存仅剩 1 件，When 两个用户几乎同时下单购买该 SKU，Then 只有 1 笔订单扣减成功，另 1 笔提示库存不足（并发压测验证不超卖）
- [ ] Given 用户下单后未支付超时，When 订单自动关闭，Then 预占的库存被正确释放回可售库存
- [ ] Given 运营为订单填写发货单（快递单号+承运商），When 用户在订单详情页查看，Then 能看到物流轨迹节点更新
- [ ] Given 用户对已完成订单申请退货，When 运营审核通过，Then 生成售后单并驱动退款流程，用户收到状态通知（复用消息通知能力）
- [ ] Given 用户订单已完成，When 用户提交带图评价，Then 评价经敏感词过滤后展示在商品详情页，违规内容被拦截或进入人工审核队列

### 🔵 Phase 3 — 增长与运营

**规模：L**（3 个新模块 + 1 项新中间件引入——Elasticsearch；秒杀子场景风险等级高，需独立限流设计，建议作为本期内部可选子任务而非阻塞项）

| 新增模块 | 核心内容 |
|---|---|
| `ruoyi-mall-marketing`（营销中心） | 优惠券、满减、限时折扣、秒杀（Redis 库存扣减+限流）、拼团（可选，参考拼多多模式） |
| 搜索与推荐 | 引入 Elasticsearch 做商品搜索，基础推荐（同类目/热销） |
| CMS 内容运营 | 首页楼层配置、专题页、Banner 管理（复用现有文件上传能力） |
| 消息通知中心统一化 | 整合短信/邮件/站内信/微信模板消息为统一出口（复用博客通知机制的设计） |

**验收标准**：
- [ ] Given 用户领取满 100 减 20 优惠券，When 结算页勾选使用，Then 订单金额正确扣减，且同一张券不能被重复使用或跨用户使用
- [ ] Given 秒杀活动库存 100 件，When 并发请求超过库存数量，Then 系统限流拦截超量请求，最终成交量不超过库存数（压测验证）
- [ ] Given 用户在搜索框输入关键词，When 提交搜索，Then 返回相关商品且支持按价格/销量排序，响应时间在可接受范围内（相较 MySQL LIKE 查询有明显提升）
- [ ] Given 运营在 CMS 后台配置首页 Banner 和楼层，When C 端刷新首页，Then 立即生效，无需重新发布前端代码
- [ ] Given 触发订单状态变更/发货等事件，When 消息中心分发通知，Then 用户在对应渠道（短信/站内信/小程序模板消息）收到一致的内容，且失败重试机制生效

### ⚪ Phase 4 — 平台化与规模化（视业务方向决定是否需要）

**规模：XL 及以上**（每个子项都是独立的架构级投资，建议拆分为独立立项而非合并排期；具体规模取决于第 9 节的业务方决策结果）

| 新增模块 | 触发条件 |
|---|---|
| 商家/店铺中心（多商家入驻） | 若业务方确认要做平台型电商（类淘宝/拼多多）而非纯自营 |
| 风控中心 | 交易量上升、出现刷单/欺诈信号后 |
| 财务结算中心 | 出现多商家分账/发票需求后 |
| 数据中台/BI 看板 | 需要经营决策数据支撑时 |
| 开放 API 网关 | 需要接入 App/小程序等多端或对外开放接口时 |
| 分库分表/消息队列（RocketMQ）/Seata 分布式事务 | 订单量增长到单库/本地事务成为瓶颈时再引入，避免过早架构复杂化 |

**验收标准**（触发对应子项后再细化，此处仅列出通用门槛）：
- [ ] 每个子项上线前必须有独立的压测/安全评审报告（尤其风控、支付分账相关）
- [ ] 商家中心若启用，必须先完成结算规则与分账逻辑的财务侧签字确认，再开放商家入驻

### ❌ 本次不做（Non-goals，除非业务方明确要求）

| 事项 | 原因 |
|---|---|
| 多仓智能分仓算法 | 需要真实履约数据积累，一期无 ROI |
| 多币种/多语言跨境能力（SHEIN 模式） | 只有明确跨境需求才需要，否则过度设计 |
| 直播电商 | 独立的内容/流量体系，与本次交易域改造无直接依赖 |
| 供应链金融 | 属于 Phase 4 之后的战略级投资，需单独立项 |

## 5. 关键新增系统设计要点

### 5.1 商品中心
- 核心表：`mall_category`（类目树，自关联 parent_id）、`mall_brand`、`mall_spu`、`mall_sku`（规格组合+价格+库存关联）、`mall_product_attribute`
- 与现有能力复用点：图片/视频上传直接用现有 `BlogFileController` 逻辑抽出的公共上传组件；商品详情图文可复用 AI 文案生成能力做辅助撰写

### 5.2 订单中心
- 核心表：`mall_cart`、`mall_order`、`mall_order_item`、`mall_order_log`（状态流水，用于审计和售后追溯）
- 状态机建议：`待支付 → 已支付/已取消(超时) → 待发货 → 已发货 → 已完成/售后中 → 已关闭`
- 订单超时关闭：短期可用现有 Quartz 定时任务扫描，长期建议引入延迟队列（RocketMQ 延迟消息）替代轮询

### 5.3 支付中心
- 核心表：`mall_payment_order`（支付单，与业务订单解耦，支持一单多次支付尝试）、`mall_refund_order`
- 设计原则：**支付单与订单单解耦**，通过统一支付网关模式对接微信支付/支付宝，避免每个业务模块各自接支付渠道（可参考 `ruoyi-wechat` 现有微信生态对接经验）

### 5.4 库存中心
- 核心表：`mall_inventory`（SKU × 仓库维度库存）、`mall_stock_lock`（预占记录）
- 防超卖：Redis + Lua 脚本做原子扣减，MySQL 落库做最终一致性校验；下单预占 + 支付成功扣减 + 超时释放

### 5.5 营销中心
- 优惠券：`mall_coupon`（券模板）、`mall_coupon_user`（领取记录）
- 秒杀：独立读写分离的秒杀链路（预热库存到 Redis、限流、异步下单），避免冲击主交易链路

## 6. C 端网站（前台）建设方案

| 维度 | 建议 |
|---|---|
| 定位 | 与现有管理后台 `frontend/` 分离部署，可复用当前博客 `/blog` 的"公开前台"经验，但商城建议独立子应用/独立域名（如 `mall.xxx.com`），避免与管理后台耦合发布节奏 |
| 技术栈 | 复用 Vue3 生态；若对 SEO/首屏性能有硬要求（类似 Shopify 商店），可评估 Nuxt3 SSR；若主要走小程序/App 分发，纯 SPA 即可 |
| 页面清单 | 首页 → 类目/搜索结果页 → 商品详情页 → 购物车 → 结算页（地址/优惠券/支付方式）→ 支付结果页 → 订单中心 → 个人中心（地址/收藏/会员） |
| API 边界 | 新增独立网关前缀（如 `/mall-api`），与现有管理后台 `/system-api`、博客公开接口 `PublicArticleController` 风格保持一致，权限模型区分"C端用户态"与"后台运营态" |
| 渠道复用 | 微信小程序商城可直接复用 `ruoyi-wechat` 现有小程序基础设施，减少重复对接成本 |

## 8. 技术架构关键决策点（简版 ADR 清单）

| 决策点 | 建议 | 理由 |
|---|---|---|
| 单体 vs 微服务 | 继续走"模块化单体"（Maven 多模块，参照现有 `ruoyi-blog`/`ruoyi-wechat` 模式新增 `ruoyi-mall-*`） | 当前团队规模和流量级别下，微服务的运维复杂度收益为负；等交易量真实增长到单体瓶颈再拆分 |
| 库存扣减一致性 | Redis 预扣 + MySQL 落库 + 定时对账补偿 | 平衡性能与一致性，避免一期就引入 Seata 分布式事务 |
| 消息异步化 | 一期可用现有 Quartz 轮询兜底；Phase 3/4 视订单量引入 RocketMQ/RabbitMQ | 避免过早引入中间件增加运维负担 |
| 搜索引擎 | Phase 3 引入 Elasticsearch | MySQL 模糊查询无法支撑商品搜索的排序/分词/聚合需求 |
| 支付渠道网关 | 统一支付网关抽象层，渠道可插拔 | 避免后续新增支付渠道时改动订单/业务代码 |

## 9. 需业务方明确的关键问题（决定架构复杂度和分期优先级）

1. **自营 vs 平台型**：是否需要多商家入驻？决定是否需要"商家中心"和"结算分账"能力。
2. **仓储模式**：单仓自营，还是多仓/未来海外仓（类 SHEIN）？决定库存中心和物流中心的复杂度。
3. **支付范围**：仅境内微信支付/支付宝，还是需要跨境支付（PayPal/Stripe）？
4. **库存扣减时点**：下单即扣 vs 支付成功扣？影响超卖风险与用户体验的取舍。
5. **是否需要强营销玩法**：秒杀/拼团（类拼多多）是否是必需能力，还是常规优惠券即可满足一期需求？
6. **博客模块去留**：作为内容营销/SEO 引流保留，还是与商城解耦拆分为独立系统？

以上问题建议在启动 Phase 1 开发前逐一拍板，避免中途返工。

## 10. 总结

- C 端购物网站只是整个系统的"橱窗"，真正的工程量集中在**商品、交易、库存、支付、物流、营销、售后、风控、财务、数据**这十余个后台子系统。
- 当前项目已具备可复用的**权限基座、文件上传、AI 能力、微信渠道对接、通知/审核机制**，可以显著减少 Phase 1/2 的重复建设成本。
- 建议按 Phase 1（交易闭环）→ Phase 2（履约闭环）→ Phase 3（增长运营）→ Phase 4（平台化，视需要）的顺序推进，每期都是可独立验收的里程碑，而不是一次性大爆炸式重构。
